   OpenProdoc: Gestor Documental            English 

Funciones OpenProdoc:

  • Gratuito y de código abierto (OpenSource)
  • Un gestor documental completo, con posibilidad de definir tipos de documentos y carepats, control de versiones, un sofisticado mmodelo de seguridad y con manejo del ciclo de vida de los documentos.
  • Definiciones de tipo de doucmentos y carpetas con orientación a objetos (incluyendo herencia).
  • Modificación de definiciones ya en uso.
  • Soporte de Tesauros
    • Gestión completa de tesauros multiidioma.
    • Importación y exportación en formato estándar Skos-rdf.
    • Multiples Tesauros simultaneos.
    • Validación de metadatos contra ls tesauros.
  • Puede utilizarse como applicación J2EE o como sistema personal portable (Linux, Windows, Mac), siempre con requerimientos muy bajos
  • Almacenaminto de documentos en distintos repositorios de diferentes tipos:
    • Carpetas de disco
    • BLOB
    • ftp
    • Referenciaados
    • Amazon AWS S3 (y sistemas compatibles con el API como Atmos y OpenStack))
  • Multiples bases de datos (Derby, MySQL, Oracle, DB2, PostgreSQL, SQLServer, SQLLite, HSQLDB)
  • Diferentes forms de autenticación (Ldap, DDBB, OS, sistema propio)
  • Granularidad muy fina de la administración y los permisos, permitiendo la delegación e diferentes funciones.
  • Multi-lenguaje (Inglés, Castellano, Portugués y Catalán)
  • Importación de dcouemntos digitalizados y clasificados en Kofax Capture o Abbyy Flexicapture
  • Soporte de tareas
    • programadas o asociadass a eventos.
    • Borrado y purgado de documentos y expedientes obsoletos
    • Informes de novedades y purgado
    • Importaciones y Exportaciones programadas
    • Actualización y conversion automáticas

Requerimientos de OpenProdoc:

Lo mínimo necesario para utilizar Openprooc es una base de datos y la máquina virtual Java 1.6 o superior. La base de datos contendrá la información sobre la configuración de OpenProdoc y todos los metadatos de los documentos. dependiendo de dónde se prefiera almacenar los documentos, estos pueden amacenarse en una carpeta del ordenador o en una tabla de la base de datos.

Para el acceso a los documentos, el cliente Swing (cliente "pesado" para realizar las funciones documentales y para la administración) no requiere nada más. para eel cliente Web se requiere un servidor de aplicaciones J2EE.

Aunque es imposible probar todas las posibles combinaciones de software (Sistema perativo+Java+base de datos+J2EE Server+ navegador), se ha probado diferentes combinaciones de ellos para asegurar al máximo la compaibilidad. Cualquier información sobre configuraciones comprobadas o combinaciones que presenten algún problema serán bienvenidas. El sistema se ha diseñado para usar un mínimo de intrucciones y evitar funciones propietarias, por lo que debería funcionar en la mayoría de los entornos. Que un software no se muestre aquí no quiere decir que no pueda funcionar en él.
