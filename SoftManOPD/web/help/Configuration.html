Help SMO Configuration

SMO has a default configuration of the types of folders and fields/metadata used for elements as Deparments, Products or Issues, but it's possible to change completely the default configuration. It's possible also to change the fields displayed in forms and list. This allows to have "hidden" or internal fields than can be changed in OpenProdoc but not in SMO. Even it's possible to have two or more configurations simultaneously in the same OpenProdoc installation, referencing them in the url.

The elements can be shared between different configurations, that is in example the same department type could be used in different in different configurations, always avoidding collisions, in example using different Departyments root.

If needed, new fields can be added to the definitions of the types in OpenProdoc and then added in the SMO definition of fields to maintain or to display. The fields to be displayed in the lists can be changed in order or dimensions, always maintaining coherence, that is checking that the name of the fields exists in Openprodoc.

When a user try to log in SMO, if the Url includes an OpenProdoc identifier (PDId) of the configuration document (i.e: http://localhost:8080/SoftManagementOPD/?IdConfig=a2134b344fd3 ) the application will load the specified configuration document referenced with that document PDId, otherwise (i.e.: http://localhost:8080/SoftManagementOPD/ ) SMO will load the default SMO configuration document, that has the PDId "SoftManOpdConf" and will be stored by default in OpenProdoc folder /system.

Configuration documents should be managed exactly as any other document in OpenProdoc, including versioning, security, etc.. and could be accesible or not using the OpenProdoc ACLs security. SMO will always use the last version of the document. It must be noted that SMO maintains a cache of definitions, so if you create a new version of the configuration document, you must stop and start each of the J2EE servers so the new version could be loaded.

Using different configuration documents, different folder type definitions of each elements and the OpenProdoc security, it's possible to have several virtual SMO systems in the same OpenProdoc installation.
